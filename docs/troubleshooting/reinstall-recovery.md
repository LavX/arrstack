# Reinstall and recovery

When something is wrong enough that re-running the wizard is the fastest fix. This page explains which command to use when, and exactly what survives each one.

---

## Pick the right command

| You want to... | Command | Media | state.json | Configs | admin.txt | Named volumes |
|---|---|---|---|---|---|---|
| Fix a half-finished install | `arrstack install --resume` | keep | rewritten at end | rewritten in place | rewritten | keep |
| Start completely over, same machine | `arrstack install --fresh` | keep | rewritten at end | rewritten in place (existing DBs survive) | rewritten | keep |
| Stop running, keep everything | `arrstack uninstall` | keep | keep | keep | keep | keep |
| Remove config but keep media | `arrstack uninstall --purge` | keep | keep | **deleted** (entire `config/` tree) | keep | keep (compose down only removes network) |
| Nuke the media too | See "full wipe" below | manual | manual | manual | manual | manual |

"Keep" means the file is untouched on disk. "Rewritten" means the installer overwrites the file with freshly rendered content. "Deleted" means the file or directory is removed by `rm -rf`.

Media (your storage root: torrents/, media/, extra paths) is **never** touched by the installer. You delete media yourself, deliberately, with `rm`. `state.json` and `admin.txt` also survive `uninstall --purge`; the only thing `--purge` actually removes is `<install-dir>/config/`.

---

## Command reference

### arrstack install --resume

Reads the existing `state.json`, skips the wizard for anything already answered, and re-runs the whole wiring pipeline. Use it when:

- An earlier install crashed mid-way and `state.json` exists but `install_completed_at` is missing.
- A service's config got corrupted and you want the installer to rewrite it.
- You changed something on the host (added a GPU, joined the docker group, mounted a new drive) and want the stack to pick it up.

Re-running is safe: every wiring step is idempotent. Existing API keys, admin passwords, indexers, and library entries are reused, not recreated.

```bash
arrstack install --resume
arrstack install --resume --install-dir /opt/arrstack
```

### arrstack install --fresh

Ignores any existing `state.json` and walks the wizard from scratch. Your previous config directory and media are still on disk, but the new install will generate a new `state.json` and may pick different API keys and ports.

Use when:
- You want to reconfigure from zero but keep your media files.
- `state.json` is corrupt (see "recovery from corrupted state.json" below).

```bash
arrstack install --fresh
```

### arrstack uninstall

Runs `docker compose down` and exits. Containers stop, the compose network is removed, but nothing on disk is deleted:

- `~/arrstack/state.json` stays
- `~/arrstack/docker-compose.yml` stays
- `~/arrstack/config/*` stays
- `~/arrstack/admin.txt` stays
- `storage_root` (media) stays

To bring everything back exactly as it was:
```bash
docker compose -f ~/arrstack/docker-compose.yml up -d
```

### arrstack uninstall --purge

Runs `docker compose down`, then recursively deletes `<install-dir>/config/`. Everything else survives: `state.json`, `admin.txt`, `docker-compose.yml`, and your media under `storage_root`. This is the right choice when a service's config tree is unrecoverable but your media is fine.

```bash
arrstack uninstall --purge
arrstack install --fresh        # rebuild from a clean config tree
```

`--purge` preserves `storage_root` by design. The uninstall command prints the preserved storage-root path before it exits.

### Full wipe (install dir + media)

The installer never does this for you. If you truly want every byte gone:

```bash
# 1. Capture the storage root BEFORE deleting state.json
STORAGE_ROOT=$(jq -r .storage_root < ~/arrstack/state.json)

# 2. Stop containers and remove the config tree
arrstack uninstall --purge

# 3. Delete everything that survived purge (state.json, .env, admin.txt, compose file)
rm -rf ~/arrstack

# 4. Delete the media root you saved in step 1
rm -rf "$STORAGE_ROOT"

# 5. Remove any arrstack-scoped named volumes
docker volume ls -q | grep ^arrstack_ | xargs -r docker volume rm
```

Run these commands one at a time. There is no undo.

## What lives where

```
~/arrstack/
├── state.json               # installer source of truth (zod-validated, 0600)
├── docker-compose.yml       # regenerated by install/update
├── .env                     # secrets: PUID, PGID, ENCRYPTION_KEY, VPN creds
├── admin.txt                # seeded admin credentials (0600)
├── config/                  # per-service configs, bind-mounted into containers
│   ├── qbittorrent/
│   ├── sonarr/
│   ├── radarr/
│   ├── prowlarr/
│   ├── bazarr/config/config.yaml
│   ├── jellyfin/
│   ├── jellyseerr/
│   ├── trailarr/
│   ├── caddy/
│   └── recyclarr/
└── data/                    # DEFAULT storage_root (wizard default)
    ├── torrents/{tv,movies,music,books}/
    └── media/{tv,movies,music}/
```

Other likely storage roots (you chose these yourself in the wizard):
- `/srv/arrstack/` (needs sudo)
- `/mnt/nas/arrstack/` (NAS-backed)
- paths under `extra_paths[]` if you added multiple drives

## Recovery scenarios

### Half-finished install

You ran `arrstack install`, it errored midway, and now `docker compose ps` shows some containers up and some missing.

```bash
jq -r 'if .install_completed_at then "complete" else "incomplete" end' ~/arrstack/state.json
# "incomplete" -> just resume
arrstack install --resume
```

If resume fails at the same step, run `arrstack doctor` and look at the specific failure. The "Container status" section pinpoints which service is stuck.

### Corrupted state.json

`arrstack doctor` or `install --resume` throws a zod validation error. `state.json` is not parseable or is missing required fields.

First, back it up:
```bash
cp ~/arrstack/state.json ~/arrstack/state.json.broken
```

Two paths:

1. **Hand-fix it.** Check the zod schema in `src/state/schema.ts` to see required keys. Common fixes: missing `schema_version: 1`, `services_enabled` not an array, `gpu.vendor` outside the allowed set.

2. **Regenerate it.** Run a fresh install, using the old `state.json.broken` as a reference for wizard answers:
```bash
arrstack install --fresh
# answer the wizard with values from state.json.broken
# existing configs and media are picked up automatically if paths match
```

Because container configs (`~/arrstack/config/*`) hold authoritative copies of API keys and service settings, a fresh install over the same install dir usually slots right in without losing anything meaningful. Seeded admin passwords get regenerated; old ones in `admin.txt.broken` still work until the installer resets them.

### Wiped config directory, media intact

Happens when someone runs `rm -rf ~/arrstack/config` but your `storage_root` lives elsewhere.

```bash
arrstack install --resume
```

The installer rebuilds every config from the catalog and wiring. API keys for Sonarr/Radarr/Prowlarr come back as new values and get written to `state.json.api_keys`. Jellyfin and Jellyseerr bootstraps re-run; your admin credentials in `admin.txt` still apply.

Your media is untouched, so libraries re-scan without data loss.

### Wrong install dir picked during wizard

You accepted `~/arrstack` but meant `/srv/arrstack`.

```bash
arrstack uninstall --purge                       # removes ~/arrstack/config
rm -rf ~/arrstack                                # drop state.json, compose, admin.txt
sudo mkdir -p /srv/arrstack
sudo chown -R $USER:$USER /srv/arrstack
arrstack install --fresh --install-dir /srv/arrstack
```

### Docker was reinstalled or the daemon reset

Containers are gone but volumes, configs, and media still exist.

```bash
arrstack install --resume
```

Compose recreates the network and containers from the same config tree.

### Stack works, but one service is wedged

Do not reinstall the whole stack. Target one service:

```bash
export COMPOSE="docker compose -f ~/arrstack/docker-compose.yml"
$COMPOSE stop sonarr
$COMPOSE rm -f sonarr
$COMPOSE up -d sonarr
arrstack logs sonarr
```

If the service's config is the problem:
```bash
$COMPOSE stop sonarr
mv ~/arrstack/config/sonarr ~/arrstack/config/sonarr.bak
arrstack install --resume
# the wiring step recreates sonarr's config fresh
```

### Migrating to a new machine

1. On the old machine:
   ```bash
   arrstack uninstall
   tar czf arrstack-backup.tgz -C ~ arrstack
   ```
2. Copy `arrstack-backup.tgz` and your `storage_root` to the new machine.
3. Extract to the same path:
   ```bash
   tar xzf arrstack-backup.tgz -C ~
   ```
4. Make sure PUID/PGID match or update them in `state.json` and run `arrstack install --resume`.
5. `arrstack doctor` should report green.

## What the installer never touches

- Your media library files.
- Files under paths in `extra_paths[]`.
- Docker daemon config (`/etc/docker/daemon.json`).
- Firewall rules (UFW, firewalld, nftables).
- systemd services outside of Docker.
- `/etc/hosts`, `/etc/resolv.conf`.

If one of those needs changing, you do it by hand.
